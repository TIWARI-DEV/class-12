Data, Information & Features

Data

Raw facts and figures without context, such as numbers, text, images, etc. Data becomes meaningful only when processed and interpreted.

Example: "25, 42, 37" are raw numbers without context.

Information

Processed data that has meaning, context, and relevance. Information answers "who", "what", "where", and "when" questions.

Example: "The average temperature this week was 34.6°C" provides meaningful information.

Knowledge

The application of information to make decisions or predictions. Knowledge answers "how" questions and represents understanding.

Example: "Temperatures above 35°C may cause heat exhaustion, so outdoor activities should be limited."

Characteristics of Quality Information

Accuracy

Information must be free from errors and correctly represent the real-world scenario it describes.

Timeliness

Information must be available when needed, not outdated when used for decision-making.

Relevance

Information must be pertinent to the current context and decision-making needs.

Completeness

Information should contain all necessary details without missing critical elements.

Consistency

Information should be uniform across different parts of the database and over time.

Accessibility

Information should be easily retrievable by authorized users when needed.

Database and Its Purpose

A database is a structured collection of data that is organized to model relevant aspects of reality in a way that supports processes requiring information.

Data Organization

Structures data in a logical manner for efficient storage and retrieval

Information Management

Provides tools for creating, reading, updating, and deleting information

Knowledge Extraction

Supports complex queries and analysis to derive insights from data

Types of Database Models

1960s

Hierarchical Model

Tree-like structure with parent-child relationships where each child has only one parent.

Use Case: Early mainframe databases, file systems

1970s

Network Model

Extension of hierarchical model allowing multiple parent-child relationships (many-to-many).

Use Case: Complex manufacturing systems

1980s

Relational Model

Organizes data into tables (relations) with rows and columns, using keys to establish relationships.

Use Case: Most modern business applications

1990s

Object-Oriented Model

Stores data as objects with attributes and methods, similar to object-oriented programming.

Use Case: Complex data in engineering, multimedia

2000s

NoSQL Models

Various non-relational models including document, key-value, column-family, and graph.

Use Case: Big data, real-time web apps

Database System Architecture

The architecture of a database system is greatly influenced by the underlying computer system on which it runs. Modern database systems use a client-server architecture where database functionality is divided between clients and servers.

Three-Schema Architecture

External Level (View Level)

Describes the part of the database that specific users are interested in. Multiple views can exist for different user groups.

Example: A student view shows only course information relevant to them

Conceptual Level (Logical Level)

Describes what data is stored in the database and the relationships among the data. Represents the entire database for a community of users.

Example: Entity-relationship model of the entire university database

Internal Level (Physical Level)

Describes how the data is physically stored in the database (files, indexes, storage structures).

Example: B+ tree indexes, hash tables, storage allocation

Database System Components

Query Processor

Translates queries into low-level instructions the database manager understands. Includes DDL interpreter, DML compiler, and query evaluation engine.

Storage Manager

Provides the interface between the low-level data stored in the database and the application programs and queries submitted to the system. Includes components like buffer manager, file manager, and transaction manager.

Transaction Manager

Ensures the database remains in a consistent (correct) state despite system failures and concurrent transaction executions. Responsible for ACID properties.

Data Independence

Logical Data Independence

The capacity to change the conceptual schema without having to change external schemas or application programs. Changes like adding new entities, attributes, or relationships can be made without rewriting existing applications.

Physical Data Independence

The capacity to change the internal schema without having to change the conceptual schema. Changes like using different file organizations, storage structures, or devices can be made without affecting the logical structure.

Integrity Constraints

Integrity constraints are rules that maintain the accuracy and consistency of data in a database. They prevent invalid data entry and maintain relationships between tables.

Domain Constraints

Ensure that only valid data within a specific domain is entered into a column.

CREATE TABLE Students (

Age INT CHECK (Age >= 18 AND Age <= 65),

Grade CHAR(1) CHECK (Grade IN ('A', 'B', 'C', 'D', 'F'))

);

Entity Integrity

Ensures each table has a primary key that is unique and not null.

CREATE TABLE Customers (

CustomerID INT PRIMARY KEY,

Name VARCHAR(100) NOT NULL

);

Referential Integrity

Maintains consistency between related tables through foreign key relationships.

CREATE TABLE Orders (

OrderID INT PRIMARY KEY,

CustomerID INT REFERENCES Customers(CustomerID),

OrderDate DATE NOT NULL

);

Key Constraints

Define rules for primary keys, foreign keys, and unique constraints.

-- Composite primary key

CREATE TABLE Enrollments (

StudentID INT,

CourseID INT,

PRIMARY KEY (StudentID, CourseID)

);

Referential Actions

When dealing with foreign key constraints, you can specify actions to be taken when referenced data is updated or deleted:

CASCADE

Automatically update or delete child rows when parent rows are updated or deleted.

FOREIGN KEY (DeptID) REFERENCES Departments(ID)

ON DELETE CASCADE

SET NULL

Set foreign key values to NULL when the referenced row is deleted.

FOREIGN KEY (ManagerID) REFERENCES Employees(ID)

ON DELETE SET NULL

RESTRICT

Prevent deletion of referenced rows (default behavior in most systems).

FOREIGN KEY (AccountID) REFERENCES Accounts(ID)

ON DELETE RESTRICT

Database Normalization

Normalization is the process of organizing data in a database to minimize redundancy and dependency by dividing large tables into smaller, more manageable tables and defining relationships between them.

First Normal Form (1NF)

A table is in 1NF if:

  • Contains only atomic (indivisible) values
  • No repeating groups or arrays
  • Each record is unique (no duplicate rows)

Before 1NF (Repeating Groups):

StudentID | Name | Courses

1 | John | Math, Science, History

After 1NF:

StudentID | Name | Course

1 | John | Math

1 | John | Science

1 | John | History

Second Normal Form (2NF)

A table is in 2NF if:

  • It's already in 1NF
  • All non-key attributes are fully functionally dependent on the entire primary key
  • No partial dependency (for composite keys)

Before 2NF (Partial Dependency):

OrderID | ProductID | ProductName | Quantity

After 2NF:

Orders: OrderID | ProductID | Quantity

Products: ProductID | ProductName

Third Normal Form (3NF)

A table is in 3NF if:

  • It's already in 2NF
  • No transitive dependency (non-key attributes depend only on the primary key)

Before 3NF (Transitive Dependency):

StudentID | Name | Department | DeptHead

After 3NF:

Students: StudentID | Name | DepartmentID

Departments: DepartmentID | Department | DeptHead

Database Transactions

A transaction is a sequence of operations performed as a single logical unit of work. A transaction must be atomic, consistent, isolated, and durable (ACID properties).

Atomicity

Ensures that all operations within a transaction are completed successfully; otherwise, none are. If any part fails, the entire transaction fails.

BEGIN TRANSACTION;

UPDATE Accounts SET Balance = Balance - 100 WHERE AccountID = 1;

UPDATE Accounts SET Balance = Balance + 100 WHERE AccountID = 2;

COMMIT; -- Both updates succeed or both fail

Consistency

Ensures that a transaction brings the database from one valid state to another, maintaining database invariants.

-- Total balance across all accounts should remain constant

-- Any transaction must maintain this invariant

Isolation

Ensures that concurrent execution of transactions leaves the database in the same state as if transactions were executed sequentially.

SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;

-- Ensures highest isolation level

Durability

Ensures that once a transaction has been committed, it will remain committed even in the event of a system failure.

-- Committed transactions are written to non-volatile storage

-- Survives power failures or system crashes

Transaction States

Active

The initial state where the transaction is executing.

Partially Committed

After the final operation has executed but before commit.

Committed

After successful completion and system has ensured updates persist.

Failed

After discovery that normal execution can no longer proceed.

Aborted

After the transaction has been rolled back and database restored.

Concurrency Control

Concurrency control ensures correct results for concurrent operations while maximizing performance. Common techniques include:

Locking

Prevents multiple transactions from accessing the same data simultaneously.

-- Shared (read) lock vs. Exclusive (write) lock

Timestamp Ordering

Assigns timestamps to transactions and ensures they execute in timestamp order.

Optimistic Concurrency

Assumes conflicts are rare and checks for them only at commit time.

Multiversion Concurrency

Maintains multiple versions of data items to allow read operations without blocking.

Centralized vs Distributed Databases

Centralized Database

A database located, stored, and maintained at a single location (a single computer or server).

Characteristics:

  • Single database file at one location
  • All users connect to this central database
  • Uniform management and administration

Advantages:

  • Simpler to design and maintain
  • Easier data integrity enforcement
  • Stronger security controls

Disadvantages:

  • Single point of failure
  • Performance bottleneck with many users
  • Limited scalability

Distributed Database

A database in which portions of the database are stored on multiple computers within a network, with processing distributed or replicated among different points.

Characteristics:

  • Data is distributed across multiple locations
  • Can be homogenous or heterogeneous
  • Data may be partitioned or replicated

Advantages:

  • Improved performance through parallel processing
  • Increased reliability and availability
  • Better scalability

Disadvantages:

  • Complex design and implementation
  • Higher hardware and software costs
  • More difficult to maintain data consistency

Database Security

Database security encompasses all measures taken to protect the database from unauthorized access, misuse, alteration, or destruction. It's a critical aspect of database administration that ensures confidentiality, integrity, and availability of data.

Database Security Challenges

Unauthorized Access

Preventing users from accessing data beyond their privileges.

Critical

SQL Injection

Malicious SQL code inserted into input fields to manipulate the database.

Critical

Data Leakage

Unauthorized transmission of data from within an organization to an external destination.

High

Loss of Data Integrity

Data becoming invalid or corrupted due to errors or malicious actions.

High

Denial of Service

Attacks that make database resources unavailable to legitimate users.

High

Insider Threats

Malicious actions by authorized users with legitimate access.

Critical

Database Security Measures

Authentication

Verifying the identity of users before granting access. Methods include passwords, biometrics, multi-factor authentication.

Authorization

Granting specific privileges to users based on roles (RBAC) or attributes (ABAC).

Encryption

Protecting data at rest (storage) and in transit (network) using cryptographic algorithms.

Auditing

Tracking and logging database activities for security monitoring and forensic analysis.

Backup and Recovery

Regular backups and tested recovery procedures to protect against data loss.

Network Security

Firewalls, VPNs, and intrusion detection systems to protect database servers.

SQL Injection Protection

Prepared Statements

Use parameterized queries to separate SQL code from data.

-- Vulnerable:

query = "SELECT * FROM users WHERE username = '" + username + "'";

-- Secure:

stmt = db.prepare("SELECT * FROM users WHERE username = ?");

stmt.execute([username]);

Input Validation

Validate and sanitize all user inputs before processing.

-- Whitelist validation

if (!username.match(/^[a-zA-Z0-9_]+$/)) {

throw new Error("Invalid username");

}

Least Privilege

Database accounts should have minimal required permissions.

-- Application user should only have SELECT, INSERT, UPDATE

-- Never use sa/root accounts for applications
